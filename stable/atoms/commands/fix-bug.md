# Command: /fix-bug

结构化的 Bug 修复工作流，确保修复过程包含诊断、验证和经验沉淀。

## 触发场景

- 用户报告了 Bug 或异常行为
- 需要诊断和修复代码问题

## 执行步骤

### 1. 诊断 (Diagnosis)

不急于修改代码，先理解问题全貌：
- 复现问题（明确触发条件）
- 分析根因（是逻辑错误、边界情况、还是依赖问题？）
- 定位受影响的代码范围

### 2. 提案 (Proposal)

在修改代码前，向用户解释：
- 问题的根本原因
- 计划的修复方案
- 可能的影响范围
- 等待用户确认再动手

### 3. 修复执行 (Fix)

- 应用最小化补丁（只改必要的部分）
- 保持代码风格一致性
- 避免"顺手"重构不相关的代码

### 4. 回归检查 (Regression Check)

- 运行现有测试，确保无副作用
- 为这个 Bug 添加新的测试用例
- 如果没有测试框架，至少手动验证修复效果

### 5. 经验沉淀 (Learning Record)

创建 `docs/learnings/YYYY-MM-DD-修复名称.md`，记录：

```markdown
# 复盘：[问题描述]

## 问题根因
（深入分析，不仅仅是表面现象）

## 解决方案
（采取了什么具体措施）

## 经验总结
- 未来如何避免类似问题？
- 是否需要更新 Rules 或 Skill？
- 是否需要添加新的测试覆盖？
```
