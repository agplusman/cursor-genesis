```markdown
# System Architecture Meta-Rules

> This is the root meta-rule. Module-specific rules generated by base-rule-generator
> inherit these constraints. Changes here cascade to all modules.

## Architecture Layers

The system follows a strict layered architecture. Dependency direction is top-down only:

1. **Interface Layer** (controllers, API endpoints, DTOs)
2. **Application Layer** (use cases, orchestration, commands/queries)
3. **Domain Layer** (entities, value objects, domain services, repository interfaces)
4. **Infrastructure Layer** (persistence, external services, framework adapters)

MUST: Each layer only depends on the layer directly below it.
MUST NOT: Lower layers reference upper layers (no reverse dependencies).
MUST NOT: Interface layer directly accesses Domain layer (go through Application).

## Domain Boundaries

| Domain Type | Definition | Dependency Rule |
|:---|:---|:---|
| Core | Primary business value | May depend on Supporting/Generic, never on other Core |
| Supporting | Assists Core domains | May depend on Generic only |
| Generic | Cross-cutting utilities | No domain dependencies |

Cross-domain communication MUST go through:
- Application-layer service calls (same process)
- Events / message contracts (cross-process)

Direct import of another domain's internal classes is FORBIDDEN.

## Directory Structure Convention

```
src/modules/{module-name}/
├── api/              # Interface layer: controllers, request/response DTOs
├── application/      # Application layer: use cases, commands, queries
├── domain/           # Domain layer: entities, value objects, services
│   ├── model/
│   ├── service/
│   └── repository/   # Repository interfaces (not implementations)
├── infrastructure/   # Infrastructure: persistence, adapters
│   ├── persistence/
│   └── external/
└── config/           # Module-specific configuration
```

## Naming Standards

- Files: kebab-case (`user-service.ts`, `order-repository.java`)
- Classes: PascalCase (`UserService`, `OrderRepository`)
- Interfaces: prefix with `I` or suffix with role (`IUserRepository` / `UserRepository`)
- DTOs: suffix with `Request`/`Response`/`DTO` (`CreateUserRequest`)

## Code Generation Constraints

When AI generates code for any module:
- MUST follow the directory structure above
- MUST place files in the correct layer
- MUST NOT create cross-domain imports without explicit anti-corruption layer
- SHOULD generate interface-first (define contract, then implement)
- SHOULD include basic input validation in DTOs
```

## Usage

1. Copy the template above into `.cursor/rules/00-meta-architecture.mdc`
2. Customize architecture layers, domain boundaries, and directory structure for your project
3. This meta-rule loads in every conversation (`alwaysApply: true`)
4. Use `base-rule-generator` Mode B to generate module-specific rules that inherit from it

## Customization Points

| Section | What to customize |
|:---|:---|
| Architecture Layers | Adjust layer names and count for your stack (e.g., 3-tier vs DDD 4-layer) |
| Domain Boundaries | List your actual core/supporting/generic domains |
| Directory Structure | Match your project's package/folder conventions |
| Naming Standards | Align with team's existing naming conventions |
| Code Generation Constraints | Add project-specific rules (framework idioms, test patterns) |
