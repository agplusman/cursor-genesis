# 本体维护指南

> ODD 方法论第四步：规范本体在项目演化过程中的持续维护。
> 原则：本体是活文档，随项目演化而演化，不是一次性产出。

---

## 一、触发维护的时机

以下任一情况发生时，必须更新本体：

| 触发事件 | 需更新内容 |
|:---|:---|
| 架构讨论、设计评审得出新结论 | 对应子域 ontology-extracted.md |
| 需求变更导致概念/字段变化 | 受影响子域的本体属性 |
| 表结构、model.yaml、entity 变更 | 按校验规则执行反向校验 |
| 发现本体与实现不一致 | 以需求文档为准修正 |
| 新子域接入 | 新建本体文件，在全局域架构文档中注册 |

**不触发维护的**：纯代码重构（不改变业务语义）、UI 调整。

---

## 二、维护流程

```text
触发事件
    ├─→ 业务概念变化（新类、新属性、规则变更）
    │       → 更新对应子域的 ontology-extracted.md
    │
    ├─→ 表/模块/依赖变化
    │       → 更新技术映射文件（tables.yaml / modules.yaml）
    │
    └─→ 概念↔实现映射变化
            → 更新 mapping.yaml（若有）
```

---

## 三、版本控制约定

- 本体文件通过 Git 管理，变更有完整历史
- 重大版本变更（聚合根重命名、域拆分/合并）在文件头部记录变更日期和原因
- 不删除历史字段，改为标注 `deprecated: true` 并说明原因

---

## 四、本体健康度检查（定期）

建议每个里程碑节点执行一次：

| 检查项 | 健康标准 |
|:---|:---|
| 本体 vs 最新表结构一致 | 无字段差异 |
| 所有跨域引用有效 | 目标类均存在 |
| 无孤立属性 | 每个属性归属明确的类 |
| 业务规则与实现一致 | 规则在代码中有对应约束 |
| deprecated 字段已清理 | 距标注超过 1 个迭代的 deprecated 字段已从实现中移除 |

---

## 五、多人协作约定

- 本体文件是**设计文档**，变更需评审（非单人自行修改）
- Agent 读取本体后如发现不一致，先提出问题，不自行修改本体
- 本体变更需同步通知依赖该子域的其他子域负责人
