# 本体提取规则

> ODD 方法论第一步：规范本体提取的输入、流程、产出，确保不遗漏。
> 与 `ontology-document-guide.md` 的关系：本文档专注「从哪提取、如何不漏」；document-guide 专注「怎么写」。

---

## 一、输入要求

### 1.1 素材类型

| 素材类型 | 必选/可选 | 说明 |
|:---|:---|:---|
| **需求/架构文档** | 必选 | 需求规格说明书、架构设计文档、标书 |
| **实现层素材** | 必选 | 表结构、接口 spec、model.yaml、原型字段；用于对照，避免漏提 |
| **已有本体** | 可选 | 派生子域时引用父域本体 |

**规则**：提取前必须列出本领域所有素材，并确认实现层素材已纳入。

### 1.2 素材登记

- 新增领域素材时登记路径、用途、关键行号（若适用）
- 提取完成后标记已用于哪些本体

---

## 二、五步提取流程

| 步骤 | 动作 | 产出 |
|:---|:---|:---|
| 1. 规格说明 | 明确子域范围、不覆盖内容 | 范围段落 |
| 2. 素材扫描 | 按素材清单逐文档扫描，对自然段中的字段做**显式摘录**（不依赖记忆） | 原始字段清单（含行号/文档） |
| 3. 实现层对照 | 用表结构/接口字段做正向清单，逐项标记「需求有/无」「已提取/未提取」 | 对照表 |
| 4. 概念化 | 类、对象属性、数据属性归类 | 类表、属性表 |
| 5. 形式化 | 表格输出，并填写「来源行号/文档」便于追溯 | ontology-extracted.md |

### 显式摘录原则

- **不依赖记忆**：从文档中逐段摘录字段名，标注出处（文档+行号）
- **合并去重**：同一概念在多处出现时，合并为一条，注明所有出处
- **自然段嵌入**：需求文档常用自然段描述，需主动识别隐含字段（如「记录操作人」→ `operatorId`）

---

## 三、提取完整性清单

提取结束后，用以下清单验收：

| 检查项 | 通过标准 |
|:---|:---|
| 核心类是否识别 | 子域聚合根已定义，有独立生命周期 |
| 属性完整性 | 每个类的属性数量支撑业务含义（仅 id+name 为提取不充分） |
| 实现层对照完成 | 所有表字段均已标记「已提取/未提取」 |
| 来源可追溯 | 每个属性有文档+行号来源 |
| 跨域引用已标注 | 引用其他域的对象属性已标注目标域 |

---

## 四、常见遗漏模式

| 遗漏模式 | 原因 | 预防方法 |
|:---|:---|:---|
| 自然段隐含字段 | 只扫描表格，忽略叙述段落 | 显式摘录原则，逐段处理 |
| 实现层新增字段 | 只看需求不看实现 | 实现层对照步骤强制执行 |
| 跨域共享概念 | 只关注本域 | 提取前先读全局域架构文档 |
| 枚举值遗漏 | 属性提取了但值域没列 | 每个枚举属性必须列出已知值 |
