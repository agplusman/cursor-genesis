# 本体校验规则

> ODD 方法论第三步：校验本体的完整性、一致性和跨域对齐。
> 触发时机：实现层变更后（表结构、model.yaml、entity 变更），执行反向校验。

---

## 一、三类校验

### 1.1 完整性校验（本体内部）

| 检查项 | 通过标准 |
|:---|:---|
| 每个类有聚合根 | 子域的顶层实体已定义 |
| 属性完整性 | 每个类属性数量能支撑业务含义（不止 id+name） |
| 枚举值完整 | 所有枚举属性列出了已知值 |
| 生命周期完整 | 核心本体有状态机定义 |
| 来源可追溯 | 每个属性有来源标注 |

### 1.2 一致性校验（本体 vs 实现）

| 检查项 | 校验方式 |
|:---|:---|
| 字段名一致 | 本体属性名 vs 表字段名 / model.yaml 字段名 |
| 类型一致 | 本体数据类型 vs 表字段类型 |
| 枚举值一致 | 本体枚举值 vs 代码枚举类 |
| 跨域引用一致 | 本体对象属性 vs 外键约束 |

### 1.3 跨域对齐校验

| 检查项 | 通过标准 |
|:---|:---|
| 引用目标存在 | 对象属性引用的目标类在目标域的本体中存在 |
| 依赖方向合法 | 基础域不依赖流程域（单向依赖） |
| 共享概念一致 | 同一概念在多个域中描述不矛盾 |

---

## 二、反向校验流程

实现层变更后，执行：

```text
实现变更（表/model.yaml/entity）
    ↓
1. 列出变更涉及的字段/类
2. 在本体中查找对应属性
3. 若本体无对应 → 补充本体
4. 若本体有但不一致 → 确认以哪个为准（需求变更 → 更新本体；实现偏差 → 修正实现）
5. 若一致 → 无需操作
```

---

## 三、校验结果处理

| 结果 | 处理方式 |
|:---|:---|
| 本体缺失字段 | 补充本体属性，标注来源 |
| 本体与实现不一致 | 明确以需求为准还是以实现为准，更新落后的一方 |
| 跨域引用断裂 | 检查目标域本体是否已定义，若未定义则补充 |
| 依赖方向违反 | 重构引用关系，基础域不得引用流程域 |
