# 上下文管理指南

> 回答：怎么管理 AI 对话的上下文，避免"越聊越笨"？

## 核心认识

上下文窗口是**有限的稀缺资源**。即使窗口很大，"context rot"（召回率和注意力随 token 数量递减）仍然存在。目标不是"给 AI 更多信息"，而是找到**最小的高信号 token 集**。

## 五个实践策略

### 1. 新开 vs 续聊

| 下一个 prompt... | 决定 |
|:---|:---|
| 依赖当前对话的历史和上下文 | 续聊 |
| 是独立的新任务 | 新开对话 |

**原则**：每个逻辑任务一个会话。不确定时，新开。

### 2. 按需检索（JIT Retrieval）

不要把所有文件一股脑塞给 AI。给它**导航工具**，让它逐层发现：

```
文件夹名 → 文件名 → 文件内容
```

这与人类浏览文件系统的认知路径同构。元数据（文件路径、命名约定、时间戳）是高信号低成本的上下文来源。

**在 Cursor 中的实践**：
- 精准使用 `@Files`/`@Code`/`@Branch`/`@Past Chats`
- 避免 `@Codebase` 的无差别扫描
- 文件命名要有语义性，让 Agent 从名字就能判断相关性

### 3. 上下文压缩（Compaction）

当对话变长时，关注 Cursor 的上下文使用量指示器：
- 接近上限前，用 `/compact` 或 `/summarize` 压缩
- 压缩时**保留**：架构决策、发现的 bug、核心计划
- 压缩时**丢弃**：原始工具输出、探索性的尝试过程

### 4. Plan Mode 作为上下文压缩器

Plan Mode 的价值不只是"先想后做"：
- 计划 = 把大量需求组块化(chunking)为结构化清单
- 这本身就是一次上下文压缩——把散乱的需求变成可引用的列表
- 小任务不需要 Plan Mode，大功能/重构/跨文件修改才需要

### 5. 小步提交

频繁 `git commit` 的三重价值：
- 更清晰的 diff → AI 更容易理解变更范围
- 更安全的回滚 → 出错时可精确恢复
- 更好的 Agent 对齐 → 上下文窗口中只看到最近的变更

## 多 Agent 的上下文隔离

当任务复杂时，用 Subagent 实现上下文隔离：

- **主 Agent**：只看高层计划和 Subagent 的摘要
- **Subagent**：在干净的窗口中处理具体子任务
- Subagent 完成后返回**摘要**（不是原始过程），主 Agent 基于摘要继续

Anthropic 的实践数据：多 Agent 在复杂研究任务上比单 Agent 成功率提升 90.2%，但 Token 消耗约 15 倍。适合高价值任务。

## 长时任务的上下文管理

超出上下文窗口的任务（数小时的重构、大型特性开发）需要 Harness 模式：

1. **状态持久化**：代码写到磁盘、变更提交到 Git、维护 TODO.md
2. **分阶段执行**：先分析+计划（Initializer），再循环执行（Coding Agent）
3. **Compaction & Resume**：窗口满时暂停 → 总结历史 → 重载摘要+文件状态+TODO → 继续

在 Cursor 中，`/checkpoint` 和 `/resume` 命令可以实现这个模式。
