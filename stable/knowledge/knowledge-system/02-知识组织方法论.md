# 知识组织方法论

> 来源：原文档 R3（认知-展示映射）+ R4（节点粒度）+ R5（领域划分）合并重组
> v2.0 更新：补充操作流程、量化标准、分层模板

---

## 一、核心问题

**认知结构与展示结构的维度差**：
- 认知结构：多维度的图结构（一个知识点可以同时属于多个父节点）
- 展示结构：单维度的树结构（一个文件只能放在一个文件夹里）

**具体例证**：
脑子里的「Git submodule」同时属于：
- Git工具集
- 模块化设计
- AI模板体系
- 认知学习的组块化

但在文件目录里只能放在一个文件夹——这就是「认知和展示对不上」的根源。

---

## 二、解决方案：树+链接

### 2.1 树状结构承载核心主线

树状结构只按「核心父类」分类，不追求覆盖所有关联关系。

**原则**：
- 一级目录 = 认知里的核心主线
- 每个节点只放在一个最核心的父类下
- 关联关系交给链接机制处理

### 2.2 显式链接补全多维关联

每个节点用 Markdown 相对路径链接关联节点。

**实现方式**：
```markdown
## 🔗 关联节点
- [DDD聚合根](../ddd/aggregate.md) - 本质相似：都是边界划分
- [Git submodule](../git/submodule.md) - 应用重叠：模块化挂载
```

**链接类型**：
| 类型 | 说明 | 示例 |
|------|------|------|
| 本质相似 | 不同领域的同构概念 | 限界上下文 ↔ 微服务边界 |
| 应用重叠 | 解决类似问题的不同方案 | RR调度 ↔ 时间片轮转 |
| 层级关联 | 上下游依赖关系 | 索引节点 → 数据节点 |

**维护原则**：
- 创建节点时只写「显而易见的关联」
- 发现新关联时增量补充
- 只记录「有实际迁移价值」的关联（理解A能帮助理解B，或A的方法能用在B上）

### 2.3 工具环境适配

**设计原则**：所有规范基于标准 Markdown，确保 Obsidian 兼容。

**当前推荐**：项目内 Markdown + 显式链接

- Cursor/Claude Code 原生支持
- 项目自包含，不依赖外部工具
- 关联用相对路径，可跨文件跳转

**Obsidian 定位**：个人「母版库」

- 沉淀跨项目的通用知识
- 项目文档是「工作副本」，完成后有价值的部分回流母版
- 不强求实时同步，定期人工回流

**Obsidian 兼容性说明**：

| 规范 | Obsidian 支持 | 说明 |
|------|---------------|------|
| 标准 Markdown 链接 `[title](path.md)` | ✅ 完全支持 | 优先使用，跨工具通用 |
| Obsidian 双链 `[[文件名]]` | ✅ Obsidian 专属 | 可选，仅在 Obsidian 内使用 |
| YAML front matter | ✅ 支持 | 可用于节点元信息 |
| HTML 注释 `<!-- -->` | ✅ 忽略渲染 | 用于 AI 触发标记 |

**迁移路径**：当前 Markdown 文件可直接拖入 Obsidian vault 使用，无需转换。

## 三、主归属判定与动态调整

### 3.1 初始归属判定

**判定顺序**（优先级从高到低）：

1. **外部约束优先**：如果知识点有明确的外部约束（考试大纲、项目规范），归入对应根节点
2. **首次场景次之**：按「首次学习/使用该知识的场景」定主归属
3. **核心价值兜底**：按「该知识的核心价值在哪个领域最大」判断

### 3.2 动态调整机制

类似红黑树的再平衡，主归属不是一成不变的。

**触发调整的信号**：
- 某知识点在「非主归属」领域被引用次数明显超过主归属（如 3:1 以上）
- 发现当前主归属其实是「应用场景」而非「本质归属」
- 元认知复盘时发现归类不合理

**调整流程**：
1. 在元认知复盘时检查，不频繁调整
2. 确认调整后，移动文件到新位置
3. 更新所有指向该节点的链接
4. 在原位置可保留一个重定向说明（可选）

## 四、节点粒度控制

### 4.1 核心法则

> 一个节点、一个核心职责、一套要素模板

### 4.2 拆分与合并的量化参考

**拆分信号（应该拆）**：
- 一个节点的「应用方法」超过 5 个独立步骤
- 一个节点同时回答两个以上不同的「什么时候用」
- 节点内容超过 500 字且包含多个并列主题
- 不同部分的「触发条件」明显不同

**合并信号（应该合）**：
- 两个节点的「触发条件」完全相同
- 一个节点的内容少于 100 字且无法独立使用
- 拆分后的节点只在一个地方被引用
- 总是一起出现、一起使用

### 4.3 边界案例处理原则

**判断依据**：实际使用中是否经常单独被问到/用到

| 场景 | 处理方式 | 示例 |
|------|----------|------|
| 总是一起出现 | 合并为一个节点 | 「HTTP状态码」作为一个节点 |
| 经常单独被问 | 拆分为多个节点 | 「RR原理」「RR计算」「RR变种」分开 |
| 不确定 | 先合并，发现需要时再拆 | 遵循最小拆分原则 |

### 4.4 应用维度的分层嵌套

「应用」不是单一维度，而是分层嵌套的：

| 层级 | 应用内容 | 示例（进程调度） |
|------|----------|------------------|
| L2 索引层 | 通用解题框架 | 本章所有题目的通用方法论 |
| L2 子索引 | 题型分类方法 | RR算法的题型分类与应对 |
| L3 数据层 | 具体解题步骤 | RR基础计算题的固定步骤 |

## 五、领域划分双轨制

### 5.1 核心矛盾

知识分类存在两个目的的天然冲突：

| 目的 | 需求 | 问题 |
|------|------|------|
| 收纳与执行 | 固定位置，符合场景规则 | 强行归错类会执行乱套 |
| 关联与迁移 | 锚定已有框架，跨领域打通 | 完全单独放会形成知识孤岛 |

### 5.2 双轨制法则

> 主归属按「核心目标」定边界，解决执行不乱
> 关联层按「本质逻辑」定联动，解决知识迁移

### 5.3 独立根节点判断标准

| 判断维度 | 独立根节点 | 归现有大类 |
|----------|------------|------------|
| 核心目标 | 有独立核心目标 | 目标完全从属于现有领域 |
| 外部约束 | 有外部强约束（考纲、规范） | 无外部约束 |
| 生命周期 | 有独立结束节点 | 无独立生命周期 |

### 5.4 新知识入库流程

```
1. 识别知识类型
   ├─ 是否有明确的外部约束？（考试、项目、规范）
   │   └─ 是 → 检查是否已有对应根节点
   │         ├─ 有 → 归入该根节点
   │         └─ 无 → 创建新根节点
   └─ 否 → 按知识的核心价值归入最相关的现有领域

2. 确定层级位置
   ├─ 是第一性原理？ → L1 根节点层
   ├─ 是方法论/框架/组块？ → L2 索引层
   └─ 是具体细节/命令/步骤？ → L3 数据层

3. 补充关联（只记录有迁移价值的）
   ├─ 与哪些现有节点有「本质相似」？
   ├─ 与哪些节点有「应用重叠」？
   └─ 写入关联节点章节
```

### 5.5 具体场景分析

**安全服务系统项目**：
- 核心目标：完成公司项目交付上线
- 外部约束：必须遵守公司技术规范
- 独立生命周期：有明确的需求、开发、测试、上线节点
- **结论**：必须单独作为独立根节点

**考研408**：
- 核心目标：202X年考研上岸
- 外部约束：必须严格按考纲来学习
- 独立生命周期：有明确的备考、模考、考试节点
- **结论**：单独作为独立根节点

## 六、节点要素模板

### 6.1 基础四要素（元指导）

所有节点的通用框架，可根据层级调整：

| 要素 | 作用 | 回答的问题 |
|------|------|------------|
| 📌 定义 | 是什么 | 这个概念/工具/方法的本质 |
| ⚡ 触发条件 | 什么时候用 | 在什么场景下应该想到它 |
| 🛠️ 应用方法 | 怎么用 | 具体的操作步骤或使用方式 |
| 🚫 边界约束 | 不能怎么用 | 限制条件、常见误用、失效场景 |

### 6.2 分层变体（推荐实践）

不同层级的节点，要素侧重不同：

| 层级 | 必填要素 | 可选要素 | 说明 |
|------|----------|----------|------|
| L1 根节点 | 定义、来源/推导、适用范围 | 边界、关联 | 重「为什么」而非「怎么用」 |
| L2 索引 | 定义、触发、应用、边界 | 关联 | 完整四要素 |
| L3 数据 | 定义、用法示例 | 边界、关联 | 简化版，避免过重 |

### 6.3 标准模板

所有模板支持可选的 YAML front matter，兼容 Obsidian 和 AI 索引。

**L1 根节点模板**：

```markdown
---
title: [根节点名称]
layer: L1
domain: [所属领域]
keywords: [关键词1, 关键词2]
---

# [根节点名称]

## 📌 定义
[这个第一性原理是什么]

## 📖 来源/推导
[为什么是这个原理？从哪里得出？]

## 🎯 适用范围
[这个原理覆盖哪些场景]

## 🚫 边界
[什么情况下这个原理不适用]

## 🔗 关联节点
[与其他根节点的关联]
```

**L2 索引模板**：

```markdown
---
title: [索引节点名称]
layer: L2
domain: [所属领域]
keywords: [关键词1, 关键词2]
---

# [索引节点名称]

## 📌 定义
[是什么]

## ⚡ 触发条件
[什么时候用]

## 🛠️ 应用方法
[怎么用]

## 🚫 边界约束
[不能怎么用]

## 🔗 关联节点
[相关节点链接]
```

**L3 数据模板**：

```markdown
---
title: [数据节点名称]
layer: L3
domain: [所属领域]
keywords: [关键词1, 关键词2]
---

# [数据节点名称]

## 📌 定义
[是什么]

## 🛠️ 用法
[怎么用，含示例]

## ⚠️ 注意事项
[常见坑、边界情况]
```

## 七、通用节点管理五步法

1. **先定根节点**：核心目标/第一性原理（≤3个）
2. **再搭索引骨架**：领域→模块→子模块，配高维应用框架
3. **再填数据细节**：单一职责叶子节点
4. **补全关联链接**：显式链接还原网状认知
5. **加元认知复盘**：定期检查归类合理性、触发动态调整

## 九、版本演进与跨项目复用

> 详细架构设计见 [05-Git模板套装架构设计.md](./05-Git模板套装架构设计.md)

### 9.1 版本演进：Git 原生支持

知识节点的版本管理直接使用 Git：

| 需求 | Git 方案 |
|------|----------|
| 追踪历史 | 自动，每次 commit 即记录 |
| 标记重大版本 | `git tag v1.0` |
| 保留旧版本 | 分支隔离 或 `v1/` `v2/` 子目录 |
| 回滚 | `git checkout <commit>` |

**不需要额外设计**，Git 本身就是版本管理工具。

### 9.2 跨项目复用：套装机制

基于 atoms + packs 架构：

```
母版库/
├── atoms/           # 原子层（最小可复用单元）
└── packs/           # 套装层（场景化组合）

项目A/.template/ ──sparse-checkout──→ 母版库/packs/frontend-dev/
项目B/.template/ ──sparse-checkout──→ 母版库/packs/backend-dev/
```

**回流机制**：

```
项目内改进 → commit → push → PR 到母版库 → 合并 → 其他项目 git pull 同步
```

**项目目录建议**：

```
项目/
├── .template/           # 从母版库拉取的通用知识（sparse-checkout）
└── .local-knowledge/    # 项目特定知识（不回流）
```

## 十、AI 辅助关联发现

### 10.1 触发场景

| 场景 | 触发方式 | 输入 |
|------|----------|------|
| 新节点入库 | 手动/自动 | 单个新节点 |
| 定期扫描 | 定时任务 | 全量节点索引 |
| 问题驱动 | 用户提问 | 问题 + 相关节点 |
| 灵感记录 | 用户输入 | 零散想法 |

### 10.2 基础流程

```
输入
  ├─ 节点内容（定义、应用、边界）
  ├─ 现有节点索引（标题 + 所属领域）
  └─ 可选：用户问题/想法

    ↓ AI 处理

输出
  ├─ 推荐关联列表
  │   ├─ 本质相似：[节点A] - 理由
  │   ├─ 应用重叠：[节点B] - 理由
  │   └─ 潜在迁移：[节点C] - 理由
  └─ 置信度评分（高/中/低）

    ↓ 人工确认

写入节点的「🔗 关联节点」章节
```

### 10.3 实现方案

**方案 A：Cursor/Claude Code 内置命令（推荐起步）**

在节点文件中添加注释触发：

```markdown
<!-- @ai-discover-links -->
```

AI 读取当前节点 + 扫描知识库目录，输出推荐关联。

**方案 B：独立脚本 + AI API**

```bash
# discover-links.sh
# 1. 收集所有节点的标题和摘要（从 index.yaml）
# 2. 调用 AI API 分析关联
# 3. 输出推荐列表供人工确认
```

**方案 C：Obsidian 插件**

利用 Obsidian 的图谱视图 + AI 插件（如 Smart Connections），自动推荐关联。

### 10.4 节点索引文件

为了让 AI 能快速了解全局，维护一个索引文件：

```yaml
# atoms/index.yaml
nodes:
  - path: prompts/code-review.md
    title: 代码审查
    domain: 工程实践
    keywords: [review, PR, 质量]
    layer: L2  # 索引层

  - path: prompts/git-commit.md
    title: Git 提交规范
    domain: 工程实践
    keywords: [git, commit, 规范]
    layer: L3  # 数据层
```

**索引维护时机**：
- 新增节点时同步更新
- 可写脚本自动从文件头提取元信息

### 10.5 AI Prompt 模板

**新节点关联发现 Prompt**：

```
你是一个知识关联分析助手。

当前新节点：
- 标题：{title}
- 定义：{definition}
- 应用场景：{application}
- 所属领域：{domain}

现有知识库索引：
{index_yaml_content}

请分析这个新节点与现有节点的潜在关联，输出：
1. 本质相似（不同领域的同构概念）
2. 应用重叠（解决类似问题）
3. 潜在迁移（方法可互相借鉴）

每个关联给出：节点路径、关联类型、理由（一句话）、置信度（高/中/低）
只输出有实际迁移价值的关联，不要凑数。
```
