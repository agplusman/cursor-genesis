## 一、讨论背景

基于 01-结构化认知模型 的 3+2 架构（核心三层 + 横切维度），讨论如何在实践中组织知识。

讨论的四个核心问题：

1. 认知-展示映射机制
2. 节点粒度标准
3. 双轨制领域划分
4. 四要素是否需要调整

## 二、讨论过程与结论

### 2.1 认���-展示映射机制

**原方案**：树承载主线 + 图补全关联

**讨论要点**：

- 用户工具环境：Cursor + Markdown 为主，Obsidian 作为个人通用知识库
- Obsidian 难以跨项目复用，目前是项目内维护文档

**结论**：

- 当前推荐：项目内 Markdown + 显式相对路径链接
- Obsidian 定位为「母版库」，项目文档完成后有价值的部分回流
- 不强求实时同步，定期人工回流即可

### 2.2 主归属判定与动态调整

**原方案**：只有静态判定，无调整机制

**讨论要点**：

- 用户提出：类似红黑树的动态调整机制
- 首次场景定初始值，但后续可根据使用情况调整

**结论**：

新增「动态调整机制」：

- 触发信号：非主归属领域引用次数明显超过主归属（如 3:1）
- 调整时机：元认知复盘时检查，不频繁调整
- 调整流程：移动文件 → 更新链接 → 可选保留重定向

### 2.3 节点粒度标准

**原方案**：「应用无法拆分为独立可复用步骤」+ 「核心职责单一」

**讨论要点**：

- 原标准过于抽象，需要量化参考

**结论**：

新增量化标准：

**拆分信号**：

- 应用方法超过 5 个独立步骤
- 同时回答两个以上不同的「什么时候用」
- 内容超过 500 字且包含多个并列主题

**合并信号**：

- 触发条件完全相同
- 内容少于 100 字且无法独立使用
- 拆分后只在一个地方被引用

### 2.4 四要素调整

**原方案**：所有节点统一使用 定义、触发、应用、边界

**讨论要点**：

- L1 根节点更需要「为什么」而非「怎么用」
- L3 数据层四要素可能过重
- 用户同意：基础版作为元指导保留，分层变体作为推荐实践

**结论**：

保留基础四要素作为「元指导」，新增分层变体：

| 层级 | 必填要素 | 可选要素 |
|------|----------|----------|
| L1 根节点 | 定义、来源/推导、适用范围 | 边界、关联 |
| L2 索引 | 定义、触发、应用、边界 | 关联 |
| L3 数据 | 定义、用法示例 | 边界、关联 |

## 四、遗留问题及解决方案

### 4.1 版本演进 → Git 原生支持

**结论**：不需要额外设计，Git 本身就是版本管理工具。

- 追踪历史：每次 commit 自动记录
- 标记重大版本：git tag
- 保留旧版本：分支隔离或子目录

### 4.2 跨项目复用 → 套装机制

**结论**：基于 05-Git模板套装架构设计 的 atoms + packs 架构。

- 母版库存放通用知识（atoms + packs）
- 项目通过 sparse-checkout 拉取所需套装
- 改进通过 PR 回流母版库

**新增建议**：项目内区分 `.template/`（通用）和 `.local-knowledge/`（项目特定）

### 4.3 AI 自动发现关联 → 独立流程设计

**结论**：设计了完整的 AI 辅助关联发现机制。

核心要素：
- 触发场景：新节点入库、定期扫描、问题驱动、灵感记录
- 索引文件：`atoms/index.yaml` 存储节点元信息
- AI Prompt 模板：分析本质相似、应用重叠、潜在迁移
- 实现方案：Cursor 注释触发 / 独立脚本 / Obsidian 插件
