# 叶子节点治理框架

> 来源：05-分布式知识管理架构 讨论延伸
> 核心思想：借鉴数据治理，统一框架管理所有叶子节点，知识反哺框架形成元循环

---

## 一、设计背景

### 1.1 问题识别

05 文档的架构偏"存储+同步"，缺少：
- **内化层**：原始文档 → 结构化知识
- **输出层**：知识 → 可直接使用的交付物
- **统一治理**：每个节点各自实现，缺乏规范

### 1.2 数据治理类比

```
数据治理                    知识管理
─────────────────────────────────────────
原始库 (ODS)         →     content/raw/
清洗/标准化          →     content/cleaned/
汇聚库 (DWD)         →     knowledge/
标签/指标            →     index.yaml + tags
应用层 (ADS)         →     output/
数据血缘             →     relations.yaml
数据治理平台         →     leaf-node-framework
```

### 1.3 核心洞察

> 每个叶子节点如果都要遵照同一套流程，这一套可以提炼为独立的治理框架项目。
> 框架的设计又可以参考数据治理知识节点，形成"元循环"。

## 三、Framework 核心设计

### 3.1 核心引擎 (core/)

```
core/
├── cli.sh                       # 命令行入口
├── lifecycle.sh                 # 生命周期管理
├── hooks/                       # 钩子机制
│   ├── pre-ingest.sh
│   ├── post-transform.sh
│   └── pre-export.sh
└── lib/
    ├── logger.sh               # 日志
    ├── validator.sh            # 校验
    └── registry.sh             # 注册到上层
```

命令行使用：

```bash
# 在叶子节点目录下
lnf init                         # 初始化节点
lnf ingest                       # 执行采集
lnf transform                    # 执行转化
lnf export                       # 执行输出
lnf validate                     # 质量校验
lnf status                       # 查看状态
```

### 3.2 流水线定义 (pipelines/)

```yaml
# pipelines/standard.yaml
# 标准处理流水线

name: standard
version: 1.0

stages:
  # ========== 采集层 ==========
  ingest:
    description: 原始数据采集
    inputs:
      - type: git
        config_key: sources.git
      - type: api
        config_key: sources.api
      - type: manual
        path: content/manual/

    outputs:
      - path: content/raw/
        format: original

    quality_gates:
      - rule: not_empty
      - rule: valid_encoding

  # ========== 清洗层 ==========
  clean:
    description: 数据清洗标准化
    inputs:
      - from_stage: ingest

    processors:
      - type: dedup
        description: 去重
      - type: normalize
        description: 格式标准化
        config:
          encoding: utf-8
          line_ending: lf
      - type: extract_metadata
        description: 提取元数据
        output: content/metadata/

    outputs:
      - path: content/cleaned/

  # ========== 转化层 ==========
  transform:
    description: 知识内化
    inputs:
      - from_stage: clean

    processors:
      - type: index
        description: 建立索引
        output: knowledge/index.yaml
      - type: tag
        description: 打标签
        config:
          taxonomy: governance/taxonomy.yaml
      - type: summarize
        description: 生成摘要
        output: knowledge/summaries/
      - type: extract_concepts
        description: 概念提取
        output: knowledge/concepts/
      - type: link
        description: 建立关联
        output: knowledge/relations.yaml

    outputs:
      - path: knowledge/

  # ========== 输出层 ==========
  export:
    description: 生成交付物
    inputs:
      - from_stage: transform

    generators:
      - type: cheatsheet
        template: templates/cheatsheet.md.tpl
        output: output/cheatsheet.md
      - type: rules
        condition: domain == "工具-*"
        output: output/rules/
      - type: api
        condition: has_api_docs
        output: output/api-reference/

    outputs:
      - path: output/

    # 输出物注册到上层
    register:
      - path: output/
        to: knowledge-graph/index/outputs-registry.yaml
```

## 四、节点配置 (meta.yaml)

### 4.1 完整配置示例

```yaml
# cursor-docs/meta.yaml
# 叶子节点完整配置

# ========== 基础信息 ==========
id: cursor-docs
name: Cursor 文档库
version: 1.2.0
domain: 工具-Cursor
description: Cursor 官方文档及最佳实践

# ========== 框架配置 ==========
framework:
  source: knowledge-graph/leaf-node-framework
  version: v1.0
  pipeline: standard                # 使用标准流水线

  # 组件选择（sparse-checkout）
  components:
    - core/
    - pipelines/standard.yaml
    - templates/
    - plugins/markdown

# ========== 数据源配置 ==========
sources:
  git:
    - name: official
      url: https://github.com/getcursor/cursor
      branch: main
      path: docs/
      schedule: "0 2 * * 1"         # 每周一凌晨2点

  api:
    - name: changelog
      endpoint: https://cursor.sh/api/changelog
      schedule: daily

  manual:
    path: content/manual/
    description: 个人使用笔记

# ========== 处理器配置 ==========
processors:
  # 覆盖或扩展标准流水线的处理器
  transform:
    - type: extract_rules
      enabled: true
      config:
        source_patterns:
          - "**/*.mdc"
          - "**/rules/**"
        output: knowledge/rules/

    - type: custom
      script: scripts/extract-shortcuts.py
      output: knowledge/shortcuts.yaml

# ========== 输出配置 ==========
outputs:
  # 定义本节点的输出物
  - id: cursor-rules
    path: output/rules/
    description: 可直接使用的 Cursor Rules
    format: mdc
    usage: |
      复制到项目 .cursor/rules/ 目录下使用

  - id: cheatsheet
    path: output/cheatsheet.md
    description: Cursor 快捷键速查表
    format: markdown

  - id: summary
    path: output/summary.md
    description: 3分钟了解 Cursor
    format: markdown

# ========== 质量规则 ==========
quality:
  rules:
    - id: doc_freshness
      description: 文档新鲜度检查
      condition: last_sync < 7 days
      severity: warning

    - id: index_completeness
      description: 索引完整性
      condition: all_files_indexed
      severity: error

# ========== 生命周期 ==========
lifecycle:
  created: 2026-01-15
  status: active                    # active | archived | deprecated
  retention:
    raw: 30d                        # 原始数据保留30天
    cleaned: 90d
    knowledge: permanent

  # 依赖声明
  dependencies:
    - node: ai-papers
      reason: prompt engineering 相关内容引用
```

## 五、治理规范 (governance/)

### 5.1 分层标准

```yaml
# governance/layer-standards.yaml
# 数据分层标准（借鉴数仓分层）

layers:
  L0_RAW:
    name: 原始层
    path: content/raw/
    description: 原始数据，不做任何修改
    retention: 30d
    quality:
      - valid_encoding
      - not_corrupted

  L1_CLEANED:
    name: 清洗层
    path: content/cleaned/
    description: 清洗后的标准化数据
    retention: 90d
    quality:
      - deduped
      - normalized_format
      - has_metadata

  L2_KNOWLEDGE:
    name: 知识层
    path: knowledge/
    description: 结构化的知识资产
    retention: permanent
    quality:
      - indexed
      - tagged
      - linked

  L3_OUTPUT:
    name: 输出层
    path: output/
    description: 可直接使用的交付物
    retention: permanent
    quality:
      - documented
      - versioned
      - tested
```

### 5.2 标签体系

```yaml
# governance/taxonomy.yaml
# 统一标签分类体系

taxonomy:
  # 内容类型
  content_type:
    - concept          # 概念定义
    - tutorial         # 教程
    - reference        # 参考文档
    - example          # 示例
    - troubleshooting  # 问题排查

  # 成熟度
  maturity:
    - draft            # 草稿
    - review           # 待审核
    - stable           # 稳定
    - deprecated       # 已废弃

  # 复杂度
  complexity:
    - beginner
    - intermediate
    - advanced

  # 来源可信度
  source_trust:
    - official         # 官方
    - community        # 社区
    - personal         # 个人
    - ai_generated     # AI生成
```

### 5.3 质量规则

```yaml
# governance/quality-rules.yaml
# 质量检查规则库

rules:
  # ========== 通用规则 ==========
  common:
    - id: encoding_utf8
      description: 文件必须是 UTF-8 编码
      check: file.encoding == 'utf-8'
      severity: error
      auto_fix: true

    - id: no_sensitive_data
      description: 不包含敏感信息
      check: not contains(content, patterns.sensitive)
      severity: critical

    - id: has_metadata
      description: 必须有元数据
      check: exists(file.frontmatter) or exists(file.metadata)
      severity: warning

  # ========== 索引规则 ==========
  index:
    - id: all_files_indexed
      description: 所有文件都被索引
      check: count(files) == count(index.entries)
      severity: error

    - id: no_dead_links
      description: 索引无死链
      check: all(index.entries, exists(entry.path))
      severity: error

  # ========== 新鲜度规则 ==========
  freshness:
    - id: source_sync
      description: 数据源同步及时
      check: last_sync < config.sources.*.schedule + 1d
      severity: warning

    - id: output_rebuild
      description: 输出物及时更新
      check: output.mtime > knowledge.mtime
      severity: warning
```

### 6.2 框架更新流程

```yaml
# leaf-node-framework/meta.yaml
id: leaf-node-framework
name: 叶子节点治理框架
type: framework                    # 特殊类型

# 知识来源
knowledge_sources:
  - node: data/data-governance
    watch: output/patterns/
    on_change: trigger_review

  - node: data/software-engineering
    watch: output/best-practices/
    on_change: trigger_review

# 框架更新流程
update_process:
  1_detect:
    description: 检测知识源变更
    trigger: knowledge_sources.*.on_change

  2_review:
    description: 人工审核变更
    assignee: maintainer
    output: changelog.md

  3_update:
    description: 更新框架代码
    requires: review.approved

  4_test:
    description: 在测试节点验证
    test_nodes:
      - data/cursor-docs
      - data/ai-papers

  5_release:
    description: 发布新版本
    requires: test.passed
    actions:
      - bump_version
      - update_changelog
      - notify_consumers
```

## 九、设计原则总结

| 原则 | 体现 |
|------|------|
| 统一治理 | 所有叶子节点使用同一套 framework |
| 按需裁剪 | sparse-checkout 只拉取需要的组件 |
| 分层清晰 | L0原始 → L1清洗 → L2知识 → L3输出 |
| 插件扩展 | 不同类型内容用不同 processor/generator |
| 元循环 | 数据治理知识反哺治理框架本身 |
| 入口明确 | meta.yaml 作为动态内容的"说明书" |

---

## 十、实施路径

1. **Phase 1**：设计 meta.yaml 规范，在 cursor-docs 试点
2. **Phase 2**：提炼 framework 核心（core + standard pipeline）
3. **Phase 3**：实现插件机制，支持不同类型节点
4. **Phase 4**：建立 data-governance 节点，形成元循环
5. **Phase 5**：逐步迁移现有知识到新架构
