## 一、增长模式总览

知识增长不是单一模式，而是四种模式并存：

| 模式 | 触发方式 | 核心目标 | 输出 |
|------|----------|----------|------|
| 经验沉淀 | 项目进行中/结束后 | 把实践经验结构化 | 知识节点 + 关联 |
| 目标驱动 | 新目标确立 | 补全能力缺口 | 学习计划 + 执行 |
| 机会探索 | 发现外部机会 | 快速参与并收获 | 经验记录 + 关联 |
| 持续追踪 | 定时/手动触发 | 跟踪领域动态 | 学习建议 + 节点更新 |

```
                    ┌─────────────────────────────────────┐
                    │           知识增长入口               │
                    └─────────────────────────────────────┘
                                     │
          ┌──────────────┬──────────┴──────────┬──────────────┐
          ↓              ↓                     ↓              ↓
    ┌──────────┐  ┌──────────┐          ┌──────────┐  ┌──────────┐
    │ 经验沉淀 │  │ 目标驱动 │          │ 机会探索 │  │ 持续追踪 │
    │ Capture  │  │ Goal     │          │ Explore  │  │ Track    │
    └────┬─────┘  └────┬─────┘          └────┬─────┘  └────┬─────┘
         │             │                     │              │
         └─────────────┴──────────┬──────────┴──────────────┘
                                  ↓
                    ┌─────────────────────────────────────┐
                    │         02 知识入库流程              │
                    └─────────────────────────────────────┘
                                  ↓
                    ┌─────────────────────────────────────┐
                    │         增长循环入口检查             │
                    │  - 是否有待验证关联？               │
                    │  - 是否填补了已知缺口？             │
                    │  - 是否触发框架调整？               │
                    └─────────────────────────────────────┘
```

## 二、经验沉淀模式（Capture）

### 2.1 核心原则

> 发生时即记录，结束后再整理

不要等项目结束才沉淀，而是：
- **进行中**：随时记录关键决策、踩坑、灵感（可以粗糙）
- **结束后**：复盘整理，用四要素重写，入库

### 2.2 两阶段流程

**阶段一：即时记录（进行中）**

```
触发：做出重要决策 / 踩坑 / 发现新认知
  ↓
快速记录（不要求结构完整）
  ├─ 什么场景？
  ├─ 做了什么决策？为什么？
  ├─ 结果如何？
  └─ 有什么启发？
  ↓
暂存到「待整理」区域
```

**阶段二：复盘整理（结束后）**

```
触发：项目结束 / 阶段性里程碑
  ↓
回顾「待整理」记录
  ↓
筛选有沉淀价值的内容
  ├─ 是否可复用？
  ├─ 是否填补了认知缺口？
  └─ 是否纠正了错误认知？
  ↓
用四要素重写 → 进入02入库流程
```

### 2.3 适用场景

| 场景 | 记录重点 |
|------|----------|
| 项目实践 | 架构决策、技术选型、踩坑记录 |
| AI研发规章制定 | 规则设计思路、执行效果、迭代原因 |
| AI系统设计 | 设计权衡、性能优化、失败案例 |
| 工作经验 | 协作模式、沟通技巧、管理心得 |

## 三、目标驱动模式（Goal-driven）

### 3.1 核心原则

> 从目标倒推能力缺口，制定学习计划

### 3.2 四步流程

```
1. 明确目标
   ├─ 目标是什么？（如：202X年考研上岸）
   ├─ 截止时间？
   └─ 成功标准？

2. 拆解能力链
   ├─ 达成目标需要哪些能力？
   └─ 每个能力需要哪些知识支撑？

3. 定位缺口（四象限分析）
   ├─ 🔵 已知已会：作为锚点
   ├─ 🟢 已知未会：主要学习目标
   ├─ 🟡 未知已知：通过关联激活
   └─ 🔴 未知未知：通过AI推演发现

4. 制定计划并执行
   ├─ 按优先级排序
   ├─ 分配到时间节奏（见第七章）
   └─ 执行 → 沉淀 → 更新缺口状态
```

### 3.3 当前目标清单（示例）

| 目标 | 类型 | 截止时间 | 状态 |
|------|------|----------|------|
| 计算机考研/管理学读研 | 学历提升 | 202X年 | 规划中 |
| 个人公司运营 | 商业能力 | 持续 | 进行中 |
| toG行业发展 | 行业洞察 | 持续 | 进行中 |
| 黑客松/A2A参与 | 技术探索 | 机会驱动 | 待机 |

## 四、机会探索模式（Explore）

### 4.1 核心原则

> 快速评估，轻量参与，重点沉淀

机会是不确定的，不要过度规划，但要确保收获能沉淀。

### 4.2 三步流程

```
1. 快速评估（10分钟内决定）
   ├─ 与现有目标的关联度？
   ├─ 投入产出比？
   └─ 时间窗口是否允许？

2. 轻量参与
   ├─ 设定最小投入边界
   └─ 边参与边记录（同经验沉淀模式）

3. 收获沉淀
   ├─ 参与结束后立即整理
   ├─ 提炼可复用的认知
   └─ 建立与现有知识的关联
```

### 4.3 适用场景

- 黑客松、编程比赛
- A2A、开源项目参与
- 新技术/工具试用
- 行业活动、技术分享

## 五、持续追踪模式（Track）

### 5.1 核心原则

> AI帮你筛选，告诉你该学什么，而不是把所有内容都推给你

### 5.2 追踪源配置

| 领域 | 追踪源 | 更新频率 | 说明 |
|------|--------|----------|------|
| 学术前沿 | arXiv、Nature、特定期刊 | 每周 | 周末学习 |
| 技术应用 | GitHub trending、技术博客、行业报告 | 每日/每周 | 工作日晚上 |
| 政策法规 | 政府网站、行业协会 | 每月/重大发布 | 支持手动触发 |
| 数学/哲学 | 经典书籍、课程、论文 | 按学习计划 | 长期积累 |

### 5.3 追踪流程

```
1. 定时/手动拉取新内容
   ├─ 定时：按配置频率自动执行
   └─ 手动：看到新闻/热点时主动触发

2. AI筛选与摘要
   输入：
   ├─ 新内容
   └─ 你的知识图谱索引（让AI知道你已经会什么）

   输出：
   ├─ 与现有知识的关联点
   ├─ 值得深入学习的内容（优先级排序）
   ├─ 可能需要更新的现有节点
   └─ 学习建议：[领域] [标题] [为什么值得学] [预计时间]

3. 人工确认
   ├─ 确认学习 → 进入目标驱动模式
   ├─ 直接入库 → 进入02入库流程
   └─ 忽略 → 标记为已处理

4. 增量更新现有节点
   ├─ 新内容补充现有节点 → 建议更新内容
   └─ 新内容推翻现有节点 → 触发反思迭代
```

### 5.4 AI筛选Prompt模板

```
你是知识追踪助手。

我的知识图谱索引：
{index_yaml_content}

本周新内容：
{new_content_list}

请帮我筛选：
1. 与我现有知识有关联的内容（说明关联点）
2. 值得深入学习的内容（按优先级排序，说明理由）
3. 可能需要更新的现有节点（说明哪个节点、为什么）

输出格式：
## 学习建议
| 优先级 | 领域 | 标题 | 为什么值得学 | 预计时间 |
|--------|------|------|--------------|----------|

## 节点更新建议
| 现有节点 | 更新原因 | 新内容来源 |
|----------|----------|------------|
```

## 六、关联关系标准化

### 6.1 关联类型（简化版）

只保留两类，降低决策成本：

| 类型 | 定义 | 生命周期 |
|------|------|----------|
| 🔴 持久型 | 底层原理互通，长期有效 | 永久 |
| 🟡 临时型 | 待验证/场景相关 | 默认2周，到期必须处理 |

### 6.2 升级机制

```
临时关联创建
  ↓
到期检查（2周）
  ├─ 已验证有效 → 升级为持久型
  ├─ 未验证 → 延期1周 或 删除
  └─ 项目结束时 → 批量复盘
      ├─ 有复用价值 → 升级为持久型
      └─ 仅限当前项目 → 归档或删除
```

### 6.3 关联三元组模型

```
【源节点】→【关联类型 + 关联理由】→【目标节点】
```

示例：
```markdown
## 🔗 关联节点
- 🔴 [DDD聚合根](../ddd/aggregate.md) - 本质相似：都是边界划分
- 🟡 [项目X的模块设计](../projects/x/module.md) - 场景应用：当前项目参考
```

### 6.4 关联避坑三铁律

1. **无价值关联禁止准入**：不是所有相关都值得记录
2. **临时关联到期必处理**：不允许无限期搁置
3. **升级需要验证证据**：不能凭感觉升级为持久型

## 七、学习节奏与生活习惯对齐

### 7.1 周节奏

| 时间 | 学习内容 | 对应模式 |
|------|----------|----------|
| 工作日晚上 | 工业技术应用、项目相关 | 持续追踪（技术应用类） |
| 周六 | 学术论文、理论学习 | 持续追踪（学术类）/ 目标驱动 |
| 周日 | 复盘整理、计划调整 | 经验沉淀（复盘）/ 反思迭代 |

### 7.2 临时查询

工作中随时可查，不受节奏限制：
- 从知识图谱中检索相关节点
- AI辅助匹配（输入问题，输出相关知识）
- 查询结果如有新发现 → 记录为临时关联待验证

### 7.3 执行建议

| 动作 | 频率 | 时长 | 说明 |
|------|------|------|------|
| 即时记录 | 随时 | 5分钟 | 发生时快速记录 |
| 追踪内容浏览 | 每日/每周 | 15-30分钟 | 看AI筛选结果 |
| 深度学习 | 周末 | 2-4小时 | 论文/课程/复现 |
| 复盘整理 | 每周日 | 1小时 | 整理本周记录 |
| 月度复盘 | 每月末 | 2小时 | 目标进度、框架调整 |

## 八、反思迭代机制

### 8.1 触发条件

满足任一即触发反思：

| 触发条件 | 来源 | 示例 |
|----------|------|------|
| 新旧知识冲突 | 持续追踪 | 新论文推翻了现有结论 |
| 同类问题反复出现 | 经验沉淀 | 同一个坑踩了三次 |
| 目标完成或失败 | 目标驱动 | 考研结束，复盘策略 |
| 外部约束变化 | 持续追踪 | 考纲更新、政策变化 |
| 定期复盘 | 主动触发 | 月度/季度复盘 |

### 8.2 反思范围

只分析核心骨架，不分析全部节点：
1. 根节点层（第一性原理）
2. 索引节点层（核心框架）
3. 元认知规则与体系规范
4. 核心关联模式

### 8.3 六步迭代流程

| 步骤 | 动作 | 核心目标 |
|------|------|----------|
| 1 | 识别触发原因 | 明确要解决什么问题 |
| 2 | AI辅助分析 | 定位要改什么、影响范围 |
| 3 | 设计调整方案 | 最小变更原则 |
| 4 | 小范围验证 | 至少1个实际场景验证 |
| 5 | 全量落地 | 打版本号，记录变更 |
| 6 | 关联补全 | 更新受影响的关联 |

### 8.4 成功判断标准

- ✅ 变更后能解决触发问题
- ✅ 不破坏现有关联的有效性
- ✅ 通过小范围验证

### 8.5 失败回滚条件

- ❌ 小范围验证失败
- ❌ 引入新的结构冲突
- ❌ 执行成本明显上升

### 8.6 迭代避坑五铁律

1. **最小变更原则**：每次只改必须改的
2. **先验证后全量**：所有调整先试点
3. **版本可回滚**：每次迭代打tag
4. **兼容现有体系**：不破坏工具链
5. **目标导向**：围绕核心目标

## 九、与02的衔接：增长循环入口

### 9.1 入库后自动检查

02入库流程完成后，自动进入增长循环入口：

```
02 入库完成
  ↓
增长循环入口检查
  │
  ├─ 检查1：是否有待验证关联？
  │   └─ 是 → 创建临时关联，等待验证
  │
  ├─ 检查2：是否填补了已知缺口？
  │   └─ 是 → 更新目标驱动模式的缺口状态
  │
  ├─ 检查3：是否触发框架调整？
  │   └─ 是 → 进入反思迭代流程
  │
  └─ 检查4：是否需要持续追踪？
      └─ 是 → 配置追踪源，加入追踪列表
```

### 9.2 增长循环闭环

```
新知识/新经验
  ↓
四种增长模式之一
  ↓
02 入库流程
  ↓
增长循环入口检查
  ↓
  ├─ 创建关联 → 等待验证 → 升级或删除
  ├─ 更新缺口 → 调整学习计划
  ├─ 触发反思 → 迭代框架
  └─ 配置追踪 → 持续更新
  ↓
循环
```

## 十、外来知识内化三步法

### 10.1 流程

1. **先判断是否符合核心根节点**
   - 所有外来知识必须能挂到现有框架
   - 完全不相关的 → 评估是否需要新建根节点

2. **再拆解核心本质**
   - 剥掉外壳，只提取核心逻辑
   - 如借鉴DDD只提取「限界上下文定边界」

3. **最后用四要素重写**
   - 用自己的语言重写
   - 放到对应层级位置

### 10.2 判断标准

| 判断点 | 通过条件 |
|--------|----------|
| 是否符合根节点 | 能用现有第一性原理解释 |
| 是否有迁移价值 | 能与至少1个现有节点建立关联 |
| 是否值得沉淀 | 预期会被引用/使用超过3次 |
