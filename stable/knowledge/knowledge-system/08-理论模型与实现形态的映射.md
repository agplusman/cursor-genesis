# 理论模型与实现形态的映射

> 来源：07-架构思维与应用路径 讨论延伸
> 核心思想：理论设计和实际应用形态不是一对一的；Human-in-the-loop 在认知系统中的角色定位

---

## 一、核心洞察

### 1.1 理论与实现的非对称映射

理论模型是"概念空间"，实现形态是"物理空间"。两者之间：
- 不是一对一映射
- 一个理论层可能被多种机制实现
- 一个实现机制可能承载多个理论层
- 有些层是显式的，有些是隐式的

### 1.2 当前系统的映射分析

| 理论层 | 实现形态 | 显式/隐式 | 承载机制 |
|--------|----------|-----------|----------|
| L1 根节点 | 领域划分、核心原则 | **隐式** | Git 仓库结构 + domains.yaml |
| L2 索引层 | 节点注册、关联关系 | **隐式** | Git 管理 + registry.yaml |
| L3 数据层 | content/knowledge/output | **显式** | 文件系统 + pipeline 流程 |
| D1 关联 | relations.yaml + 跨节点引用 | **显式** | 配置文件 + sparse-checkout |
| D2 元认知 | 人的思考 + AI 对话 | **隐式** | Human-in-the-loop |

```
理论模型                    实现形态
─────────────────────────────────────────────────────
L1 根节点层    ─┐
               ├──→  Git 仓库结构（隐式承载）
L2 索引层     ─┘

L3 数据层     ────→  文件系统 + Pipeline（显式结构）

D1 关联维度   ────→  配置文件 + Git 机制（半显式）

D2 元认知     ────→  人 + AI 对话（隐式触发）
```

## 二、隐式承载的价值

### 2.1 Git 如何隐式承载 L1/L2

我们没有专门为 L1/L2 写代码，但 Git 天然提供了：

| Git 机制 | 承载的理论功能 |
|----------|---------------|
| 仓库边界 | L1 领域划分 |
| 目录结构 | L2 知识组块 |
| submodule | L2 节点注册 |
| commit history | 知识演化追溯 |
| branch | 知识版本/变体 |
| .gitmodules | 依赖关系声明 |

**洞察**：好的基础设施（Git）本身就蕴含了某种"认知结构"，我们只是把它显式化、命名化了。

### 2.2 隐式 vs 显式的权衡

| 特性 | 隐式承载 | 显式结构 |
|------|----------|----------|
| 优点 | 轻量、自然、无额外维护 | 清晰、可查询、可自动化 |
| 缺点 | 不易发现、难以检索 | 需要维护、可能过度设计 |
| 适用 | 稳定的、底层的结构 | 需要操作的、动态的结构 |

当前设计的选择：
- L1/L2 稳定，用 Git 隐式承载
- L3/D1 需要操作，显式结构化
- D2 本质上无法完全显式化

## 三、Human-in-the-loop：人在认知系统中的角色

### 3.1 D2 元认知为什么必须有人

```
D2 元认知的三个组成：
├── 元认知知识：知道框架是什么 → AI 可以辅助
├── 元认知监控：判断框架是否适用 → 需要人的判断
└── 元认知调节：决定如何调整框架 → 需要人的决策
```

AI 能做的：
- 提供关联、总结、推导
- 回答"这个框架在什么情况下失效"
- 生成备选方案

AI 不能做的：
- 判断"当前情境是否属于失效情况"
- 决定"要不要调整框架"
- 承担调整的后果

### 3.2 人的触发机制

当前系统中，D2 元认知的触发方式：

| 触发方式 | 示例 | 效果 |
|----------|------|------|
| 反思性提问 | "我们在做什么？" | 激发 AI 总结当前状态 |
| 类比联想 | "这跟数据治理是不是类似？" | 激发 AI 建立跨域关联 |
| 边界探索 | "这个设计能兼容什么场景？" | 激发 AI 推演可能性 |
| 质疑挑战 | "这样设计有什么问题？" | 激发 AI 暴露盲点 |

**关键洞察**：人的提问本身就是元认知的触发器。问题的质量决定了 AI 输出的质量。

### 3.3 Human-in-the-loop 的角色模型

```
┌─────────────────────────────────────────────────────────────┐
│                    认知系统运行模型                          │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                    AI Agent                          │   │
│  │                                                      │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐          │   │
│  │  │ 结构提炼 │  │ 关联发现 │  │ 知识推导 │          │   │
│  │  └──────────┘  └──────────┘  └──────────┘          │   │
│  │                      ↑                               │   │
│  │                      │ 响应                          │   │
│  └──────────────────────┼──────────────────────────────┘   │
│                         │                                   │
│  ═══════════════════════╪═══════════════════════════════   │
│                         │                                   │
│  ┌──────────────────────┼──────────────────────────────┐   │
│  │                      ↓                    Human      │   │
│  │                                                      │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐          │   │
│  │  │ 提问触发 │  │ 判断决策 │  │ 方向把控 │          │   │
│  │  └──────────┘  └──────────┘  └──────────┘          │   │
│  │       │                            │                 │   │
│  │       │      D2 元认知层           │                 │   │
│  │       └────────────────────────────┘                 │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 人的不可替代性

| 职责 | 为什么不可替代 |
|------|---------------|
| 提问 | 问题定义了探索方向，AI 不知道该问什么 |
| 判断 | 情境理解需要个人经验和价值观 |
| 决策 | 需要承担后果，AI 无法承担 |
| 意义赋予 | "这对我有什么用"只有人能回答 |

## 四、触发模式的设计

### 4.1 问题即触发器

既然人的提问是 D2 的触发机制，可以设计一套"元认知问题库"：

```yaml
# meta-questions.yaml
# 元认知触发问题库

reflection:  # 反思类
  - "我们现在在做什么？"
  - "这个设计解决了什么问题？"
  - "这和之前的XX有什么关系？"

connection:  # 关联类
  - "这跟XX领域是不是类似？"
  - "有没有其他领域用类似的思想？"
  - "能不能用XX来类比？"

boundary:  # 边界类
  - "这个设计能兼容什么场景？"
  - "什么情况下这个设计会失效？"
  - "最简单的用法是什么？最复杂的呢？"

challenge:  # 挑战类
  - "这样设计有什么问题？"
  - "有没有更简单的方案？"
  - "我是不是过度设计了？"
```

### 4.2 触发的时机

| 时机 | 适合的问题类型 |
|------|---------------|
| 设计初期 | 关联类（找已有知识锚点） |
| 设计中期 | 边界类（验证灵活性） |
| 设计完成 | 反思类（总结沉淀） |
| 遇到困难 | 挑战类（暴露盲点） |

## 五、与 01 文档的呼应

### 5.1 AI 四大角色在 Human-in-the-loop 中的定位

| AI 角色 | 触发方式 | 人的配合 |
|---------|----------|----------|
| 结构提炼者 | "帮我总结..." | 人提供原始素材和方向 |
| 关联发现者 | "这跟XX有什么关系..." | 人提供类比候选 |
| 知识推导者 | "基于这个，还缺什么..." | 人判断推导是否有价值 |
| 认知挑战者 | "这个设计有什么问题..." | 人决定是否采纳挑战 |

### 5.2 完整的认知循环

```
人提问 → AI 响应 → 人判断 → 人决策 → 人提问（下一轮）
  ↑                                      │
  └──────────────────────────────────────┘
              D2 元认知循环
```

## 六、实践启示

### 6.1 设计认知系统时

1. **不追求所有层都显式化**：有些层用基础设施隐式承载更自然
2. **识别人的不可替代位置**：D2 元认知必须有人参与
3. **设计好触发机制**：问题库、检查清单、定期回顾

### 6.2 使用认知系统时

1. **学会提问**：问题质量决定 AI 输出质量
2. **保持判断**：AI 的输出需要人来验证和决策
3. **定期反思**：主动触发 D2 元认知，而不是等问题出现

### 6.3 迭代认知系统时

1. **记录触发过程**：哪些问题触发了有价值的洞察
2. **沉淀问题模式**：形成可复用的元认知问题库
3. **优化人机边界**：哪些判断可以逐步交给 AI，哪些必须保留

## 七、人与 Agent 的动态边界

### 7.1 边界是动态的，不是静态的

```
人的探索区                    Agent 代理区
─────────────────────────────────────────────────
  新问题                        成熟流程
  模糊判断                      明确规则
  价值决策                      机械执行

        ←── 提炼经验 ──→
        ←── 交接流程 ──→
```

人不断探索新领域，把成熟的部分交给 Agent，自己继续探索。这是一个**持续外包**的过程。

### 7.2 交接的条件

什么时候可以交给 Agent：

| 条件 | 说明 |
|------|------|
| 流程明确 | 输入→处理→输出 可以清晰定义 |
| 判断规则化 | 决策逻辑可以写成 if-else 或评分 |
| 容错可控 | 出错的后果可接受，或有回滚机制 |
| 成本合理 | Agent 执行成本 < 人工成本 |

### 7.3 架构如何兼容动态边界

```yaml
# 流程定义中预留人工介入点
pipeline:
  stages:
    - name: ingest
      executor: agent          # Agent 执行

    - name: transform
      executor: agent

    - name: quality_check
      executor: human_review   # 人工审核（可选）
      condition: quality_score < 0.8

    - name: export
      executor: agent

    - name: publish
      executor: human_approve  # 人工批准
      condition: is_public == true
```

关键设计：
- `executor` 字段标记执行者
- `condition` 控制何时需要人工介入
- 随着信任度提高，可以逐步把 `human_*` 改为 `agent`

### 7.4 交接的记录

```yaml
# governance/handover-log.yaml
# 记录哪些流程已交接给 Agent

handovers:
  - id: cursor-docs-sync
    date: 2026-02-24
    from: human
    to: agent
    flow: ingest → clean → index
    confidence: high
    rollback: git reset

  - id: paper-summarize
    date: 2026-03-01
    from: human
    to: agent
    flow: transform.summarize
    confidence: medium
    human_review: true        # 仍需人工审核
```

## 八、系统容错设计

### 8.1 风险识别

| 风险 | 场景 | 后果 |
|------|------|------|
| Agent 计费失控 | 复杂任务触发大量 API 调用 | 成本爆炸 |
| 多层架构崩塌 | 环境复杂度超过 Agent 理解能力 | 输出混乱/死循环 |
| 数据损坏 | 错误的批量操作 | 知识丢失 |
| 依赖断裂 | 外部 API/服务不可用 | 流程中断 |

### 8.2 Git 作为保底机制

```
┌─────────────────────────────────────────────────────────────┐
│                      系统运行层                              │
│                                                             │
│   Agent 执行 → Pipeline 处理 → 输出生成                     │
│        ↓              ↓              ↓                      │
│      可能出错       可能出错       可能出错                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                           │
                           ↓ 所有变更
┌─────────────────────────────────────────────────────────────┐
│                      Git 版本层（保底）                      │
│                                                             │
│   commit → commit → commit → commit                         │
│      ↑                                                      │
│   任意时刻可回滚                                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

Git 提供的保障：
- **时间机器**：任意历史状态可恢复
- **变更追溯**：谁在什么时候改了什么
- **分支隔离**：实验性操作不影响主线
- **原子操作**：要么成功要么不变

### 8.3 防御性设计

#### 计费控制

```yaml
# framework/limits.yaml
limits:
  agent:
    max_tokens_per_run: 100000
    max_api_calls_per_day: 1000
    cost_alert_threshold: $10

  pipeline:
    max_files_per_batch: 100
    timeout_per_stage: 300s

  retry:
    max_attempts: 3
    backoff: exponential
```

#### 复杂度熔断

```yaml
# 当环境复杂度超过阈值时，停止 Agent 自动执行
circuit_breaker:
  triggers:
    - condition: nested_depth > 5
      action: pause_and_alert
    - condition: file_count > 1000
      action: require_human_confirm
    - condition: error_rate > 0.3
      action: rollback_and_stop
```

#### 渐进式执行

```bash
# 不要一次性处理所有，分批执行
lnf transform --batch-size 10 --dry-run  # 先预览
lnf transform --batch-size 10            # 确认后执行
```

### 8.4 恢复策略

| 故障类型 | 恢复策略 |
|----------|----------|
| Agent 输出错误 | git reset 到上一个 commit |
| Pipeline 中断 | 从 .state/checkpoint 恢复 |
| 数据损坏 | git checkout 特定文件 |
| 全面崩溃 | git clone 重新开始 |

### 8.5 最小可用状态

即使所有自动化都失效，系统仍然可用：

```
最小可用状态 = Git 仓库 + 文件系统 + 人工操作

- 文档还在（content/）
- 历史还在（.git/）
- 人可以手动读写
- 等 Agent 恢复后再自动化
```

这就是为什么选择 Git + 文件系统作为底层：**它们是最稳定、最可控、最可恢复的基础设施**。

## 九、设计原则总结

| 原则 | 体现 |
|------|------|
| 动态边界 | 人与 Agent 的分工可以随时间调整 |
| 渐进交接 | 成熟一个流程，交接一个流程 |
| 保底机制 | Git 版本控制兜底所有风险 |
| 熔断保护 | 复杂度/成本超限时自动停止 |
| 最小可用 | 即使自动化全挂，手动仍可操作 |
