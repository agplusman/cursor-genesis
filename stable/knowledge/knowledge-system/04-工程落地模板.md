# 工程落地模板

> 来源：基于 01-03 推演的工程化落地方案
> 核心思想：主心骨（通用原则）+ 变体按需推导

---

## 一、通用主心骨

> 本章是从 01-03 推演的通用原则，不绑定任何具体应用场景。
> 所有变体都从这里出发，结合场景约束推导生成。

### 1.1 目录结构原则

**来源**：01 的 3+2 架构（L1/L2/L3 + D1 关联 + D2 元认知）

| 原则 | 说明 | 来源 |
|------|------|------|
| 层级映射 | 目录结构应能映射 L1/L2/L3 三层 | 01-二 |
| 关联内嵌 | 关联不是独立目录，而是节点内的章节 | 01-2.4 |
| 元认知横切 | 元认知是监控维度，不是独立层级 | 01-2.5 |
| 单一主归属 | 每个节点只放在一个位置，关联用链接 | 02-三 |

**目录骨架**：

```text
[知识体系根目录]/
├── [根节点1]/           # L1: 第一性原理
│   ├── [索引节点]/      # L2: 组块/框架
│   │   └── [数据节点]   # L3: 具体细节
│   └── ...
├── [根节点2]/
│   └── ...
└── _meta/               # 元认知：体系规范、迭代日志
    ├── 体系规范.md
    └── 迭代日志.md
```

### 1.2 节点格式原则

**来源**：02 的分层变体（6.2）

| 层级 | 必填要素 | 侧重点 |
|------|----------|--------|
| L1 根节点 | 定义、来源/推导、适用范围 | 为什么（第一性原理）|
| L2 索引 | 定义、触发、应用、边界 | 什么时候用、怎么用 |
| L3 数据 | 定义、用法示例 | 具体怎么做 |

**通用格式骨架**：

```markdown
---
title: [节点名称]
layer: L1/L2/L3
domain: [所属领域]
---

# [节点名称]

## 📌 定义
[是什么]

## [层级特定章节]
[根据层级填写不同内容]

## 🔗 关联节点
- 🔴 [持久关联](路径) - 关联理由
- 🟡 [临时关联](路径) - 关联理由 | 到期日期
```

### 1.3 AI 协作原则

**来源**：01 的 AI 四角色（四）

| 角色 | 方向 | 使用时机 |
|------|------|----------|
| 结构提炼者 | 输入→结构 | 新领域入门、整理混沌信息 |
| 关联发现者 | 结构↔结构 | 新节点入库、跨域迁移 |
| 知识推导者 | 结构→缺口 | 目标驱动学习、能力盘点 |
| 认知挑战者 | 结构→边界 | 框架验证、避免过度自信 |

**协作模式**：

```text
人负责：定义根节点、判断关联价值、决定是否采纳
AI负责：提炼结构、发现关联、推导缺口、挑战边界
```

### 1.4 检查流程原则

**来源**：02 的入库流程（五）+ 03 的增长循环（九）

**新节点入库检查**：

```text
1. 主归属判定
   ├─ 有外部约束？→ 归入对应根节点
   ├─ 无约束？→ 按首次场景或核心价值判定
   └─ 完全不相关？→ 评估是否新建根节点

2. 层级定位
   ├─ 第一性原理？→ L1
   ├─ 方法论/框架？→ L2
   └─ 具体细节？→ L3

3. 粒度检查
   ├─ 应用方法 >5 步？→ 考虑拆分
   ├─ 内容 <100 字？→ 考虑合并
   └─ 单一职责？→ 通过

4. 关联补全
   └─ 有迁移价值的关联？→ 写入关联章节
```

**增长循环检查**（入库后）：

```text
├─ 有待验证关联？→ 创建临时关联
├─ 填补了已知缺口？→ 更新缺口状态
├─ 触发框架调整？→ 进入反思迭代
└─ 需要持续追踪？→ 配置追踪源
```

### 1.5 核心设计原则

> **目的驱动**：这个东西是为了什么？
> **用途驱动**：能用来做什么？
> **来源追溯**：应该从现有什么知识推导出来？
> **动态适配**：不同场景/标准/约束，结构不同

这四条原则贯穿所有变体的推导过程。

## 二、变体推导方法

> 变体不预设，而是按需推导。
> 每次推导必须基于已入库的知识，推导过程必须记录。

### 2.1 推导原则

| 原则 | 说明 |
|------|------|
| 知识来源必须入库 | 推导依据的知识必须在管理范围内，临时引入的知识需先入库 |
| 推导过程必须记录 | 输入、决策、输出都要留痕，确保可追溯、可复现 |
| 从主心骨出发 | 每次推导都从本章原则开始，不在已有变体上魔改 |
| 变体可增删 | 应用场景变化时，变体可以废弃或重新推导 |

### 2.2 推导流程

```text
1. 明确「要做的事情」
   ├─ 核心目标是什么？
   ├─ 产出物是什么？
   └─ 谁会使用？

2. 分析构成要素
   ├─ 涉及哪些知识领域？
   ├─ 有哪些外部约束？（工具限制/政策/标准/时间节点...）
   └─ 需要哪些原始数据？

3. 识别知识来源层次
   ├─ 指导层（L1/L2）：设计原理、范式、政策标准
   ├─ 提炼层（L3）：可复用方案、模板、组件
   └─ 原始数据层：项目代码、踩坑记录、配置文件、笔记

4. 检查知识来源是否入库
   ├─ 已入库 → 记录引用路径
   └─ 未入库 → 先走 02 入库流程，再继续推导

5. 从主心骨映射结构
   ├─ 目录结构：1.1 原则 + 场景约束 → 具体目录
   ├─ 节点格式：1.2 原则 + 场景约束 → 具体模板
   ├─ AI协作：1.3 原则 + 场景约束 → 具体提示词/工具
   └─ 检查流程：1.4 原则 + 场景约束 → 具体清单

6. 识别特殊需求
   ├─ 是否需要「元工具」（创建工具的工具）？
   ├─ 是否需要分发机制？
   ├─ 是否有动态加载需求？
   └─ 是否有多来源整合需求？

7. 输出变体文档
   └─ 包含完整的推导记录
```

### 2.3 变体文档模板

```markdown
## 变体X：[名称]

### 推导记录

#### 推导日期
YYYY-MM-DD

#### 要做的事情
- 核心目标：
- 产出物：
- 使用者：

#### 推导依据（必须是已入库的知识）
| 知识节点 | 路径 | 提供了什么 |
|----------|------|------------|
| [节点名] | [路径] | [作用说明] |

#### 外部约束
- 工具/平台限制：
- 政策/标准要求：
- 时间节点：

#### 知识来源层次分析
- 指导层（L1/L2）：
- 提炼层（L3）：
- 原始数据层：

#### 推导决策
1. 目录结构为什么这样设计？
2. 节点格式为什么这样设计？
3. 有哪些特殊需求？如何处理？

---

### 场景约束
[从推导记录中提炼的约束摘要]

### 目录结构
[具体目录设计]

### 节点格式
[具体模板]

### 检查清单
[具体检查项]
```

### 2.4 变体与知识来源的关系

```text
其他知识图谱 / 外部知识
        │
        ↓ 入库（02流程）
已入库的知识（在管理范围内）
        │
        ↓ 作为推导依据
变体推导过程（本章流程）
        │
        ↓ 生成
具体变体结构
        │
        ↓ 实践验证
反馈 → 更新知识 / 重新推导变体
```

## 三、变体示例：Cursor 协作模板

### 推导记录

#### 推导日期
2026-02-24

#### 要做的事情
- 核心目标：用 Cursor 提升开发效率，支持工作项目、个人项目、比赛、AI 辅助学习等场景
- 产出物：可复用、可分发、可回流的 Cursor 组件（rules/skills/commands/agents）
- 使用者：自己 + 团队 + 可能公开

#### 推导依据（已入库的知识）

| 知识节点 | 路径 | 提供了什么 |
|----------|------|------------|
| 3+2 架构 | 01-结构化认知模型.md#二 | L1/L2/L3 分层原则 |
| AI 四角色 | 01-结构化认知模型.md#四 | AI 协作模式 |
| 分层变体模板 | 02-知识组织方法论.md#6.2 | 节点格式原则 |
| Git 模板套装架构 | 02-知识组织方法论.md#九 | atoms + packs 分发机制 |
| 一切皆文件 | 预处理文档-重构对软件的认知.md | Git 作为文件管理系统的认知 |
| Skill 动态加载 | （待入库：Cursor 官方文档） | 元工具设计依据 |

#### 外部约束
- 工具限制：必须在 `.cursor/` 目录下；支持 rules/skills/commands/agents
- 分发机制：Git submodule + sparse-checkout 按需分发
- 回流机制：项目实战验证后 PR 回流到母版库

#### 知识来源层次分析
- 指导层（L1/L2）：AI 协作原理、Skill 设计规范、Git 哲学
- 提炼层（L3）：atoms（原子组件）、packs（套装）、meta-tools（元工具）
- 原始数据层：项目中的代码片段、配置、踩坑记录

#### 推导决策

1. **目录结构为什么这样设计？**
   - atoms/packs 分离：来自 02 的套装架构设计，支持原子复用 + 场景组合
   - meta-tools 独立：因为 Skill 支持动态加载，可以做「创建工具的工具」
   - knowledge 目录：存放指导层知识，供 meta-tools 动态引用

2. **节点格式为什么这样设计？**
   - 原子组件用 L3 简化格式：定义 + 触发 + 内容 + 示例
   - 设计指南用 L2 完整格式：定义 + 触发 + 应用 + 边界

3. **特殊需求如何处理？**
   - 元工具需求：create-component.md 作为统一入口，动态加载对应指南
   - 分发需求：Git sparse-checkout 只拉取需要的 atoms/packs
   - 回流需求：项目验证 → PR → 母版库 → 其他项目同步

### 元工具设计

**核心理念**：元工具 = 创建其他工具的工具

**统一入口 skill**：`create-component.md`

```markdown
# 创建 Cursor 组件

这是一个动态加载的 skill，根据你要创建的组件类型，加载对应的设计指南和模板。

## 支持的组件类型
1. Rule - 规则
2. Skill - 技能
3. Command - 命令
4. Agent - Agent

## 使用方式
告诉我你要创建什么类型的组件，我会：
1. 加载对应的设计指南（来自 meta-tools/guides/）
2. 加载对应的模板（来自 meta-tools/templates/）
3. 引导你完成创建

## 动态加载说明
- 创建 Rule 时 → 只加载 rule-design-guide.md + rule-template.md
- 创建 Skill 时 → 只加载 skill-design-guide.md + skill-template.md
- 不会一次性加载所有内容
```

**设计来源**：
- 02 说 skill 可以「运行到某一步时才加载对应文档」
- 所以一个 skill 可以覆盖「创建所有类型组件」这件事
- 按需加载，不浪费 context

### Git 分发与回流

**分发流程**：

```text
母版库                              项目
cursor-templates/                   项目/.cursor/
    │                                   ↑
    │ git submodule add                 │
    │ + sparse-checkout                 │
    └───────────────────────────────────┘
         只拉取需要的 atoms/packs
```

**回流流程**：

```text
项目实战验证
    ↓
提炼可复用部分
    ↓
PR 到母版库
    ↓
Review + 合并
    ↓
其他项目 git pull 同步
```

### 检查清单

**新组件创建检查**：

- [ ] 使用了元工具（create-component skill）？
- [ ] 遵循了对应的设计指南？
- [ ] 填写了完整的元信息（type/layer/tags/version）？
- [ ] 触发条件明确？
- [ ] 有使用示例？
- [ ] 关联组件已标注？

**组件回流检查**：

- [ ] 经过项目实战验证？
- [ ] 剥离了项目特定内容？
- [ ] 具有通用复用价值？
- [ ] 更新了版本号？

**套装更新检查**：

- [ ] manifest.yaml 已更新？
- [ ] 依赖的 atoms 都存在？
- [ ] 测试过 sparse-checkout 能正常拉取？
